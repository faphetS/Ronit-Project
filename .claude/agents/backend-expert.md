## Project Context

- **Server location:** `Server/` (workspace in npm monorepo)
- **Entry:** `Server/src/server.ts` (Express 5 with full middleware stack)
- **Port:** 3000 (dev via `npm run dev` from `Server/`)
- **Language:** TypeScript strict mode, ES modules (`"type": "module"`, imports must use `.js` extension)
- **Framework:** Express 5, Zod for validation, pino for logging, supertest + vitest for tests. (No JWT/auth in this project — webhooks are signature-verified instead.)

## Architecture Rules (CRITICAL — follow these)

- **Domain-driven structure:** new features go in `Server/src/domains/{name}/` with `{name}.controller.ts`, `{name}.service.ts`, `{name}.routes.ts`, `{name}.validator.ts`
- **Routes aggregator:** mount domain routes in `Server/src/routes/index.ts` (mounted at `/api`)
- **Env access:** always use `env` from `./config/env.js` — never raw `process.env`
- **Supabase:** when added, always use `supabaseAdmin` from `./config/supabase.js` — never create new clients. (Not yet present.)
- **Logging:** always use `logger` from `./config/logger.js` — never `console.log`
- **Errors:** throw `AppError` subclasses from `./lib/errors.js` — never generic `Error`
- **Validation:** use `validate({ body?, params?, query? })` middleware with Zod schemas
- **Webhook verification:** verify HMAC signatures (Meta `x-hub-signature-256`, Twilio `x-twilio-signature`, Monday challenge) BEFORE parsing — use raw-body middleware on those routes
- **Database types:** when Supabase is added, import from `Server/src/types/database.ts` (auto-generated by Supabase CLI)

## Middleware Stack (preserved by `Server/src/server.ts`)

requestId → CORS → Helmet → pino-http → body parsers (1MB limit) → cookie-parser → HPP → rate limit (/api) → routes → 404 → global error handler
